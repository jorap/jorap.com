> **TL;DR**: I scope the sales call before I quote. Two or more "dynamic" yes answers - login walls, live database, cart you host, five daily non-technical editors with no plan - and it's not my stack. I'd rather send you to WordPress honestly than take your deposit and improvise.

## The question isn't "can we build it"

It's **should this stack carry the job after launch**.

I've watched developers say yes to everything because saying no feels awkward in a first meeting. Then week two hits: the client expected five staff to edit pages like WordPress, nobody learned Git, there's no retainer, and the "simple brochure site" needs member logins by Friday.

That's not a technical failure. That's a **scope lie** that should have died in the sales call.

I run a short fit check before every quote. Not to be difficult. Because the fastest way to lose trust is to cash a deposit for a job my tools can't finish.

---

## Good fits I've actually taken

**Portfolios and case studies.** A freelance designer needed somewhere to show work in order, with context, without an algorithm shuffling it. Static, live in about a week. Done.

**Org brochures and small business presence.** Hours, location, services, team photos, contact form. Text and images on git push. Exactly what a lean static stack is for.

**Project docs and reference sites.** Content that changes occasionally, mostly written by one or two people who can learn a publish path or email me updates.

**Blogs where the owner picks a handoff model.** Git lesson, retainer batches, or a Git-based CMS when buttons beat terminal. The key is **naming that path before deposit**, not improvising at launch.

**Archives that don't need accounts.** A pastor wanted sermon audio and notes online. Brochure plus archive pages. No member login, no comments wall. Static fit.

**Sites where the "dynamic" part lives somewhere else.** A gym wanted class schedules from a booking SaaS. We embedded the vendor widget. The shell around it stayed static. A cafe owner wanted online ordering - out of scope for what I'd host; we did a menu page plus a link to the delivery app they already used.

Pick by **fruit**, not by what I want to sell. If Hugo is honest, I quote Hugo. If WordPress is honest, I say that too.

---

## Bad fits I say no to (in the room)

**E-commerce you host.** Carts, checkout, inventory, payment flows you own end to end. That's Shopify or WooCommerce territory, not a markdown repo.

**Member login and private content.** Anything behind authentication that isn't handled by a third-party tool. Static sites don't have sessions. Pretending otherwise wastes everyone's month.

**Booking engines you maintain.** If the schedule has to live in your database and update in real time, you're not buying a brochure site.

**Five non-technical editors publishing daily with no plan.** WordPress can do that with training and roles. A static repo cannot - unless someone pays for a retainer, learns Git, or we set up a CMS layer on purpose. "They'll figure it out" is not a plan.

**Jobs where two weeks without you means panic.** If nobody on their side can publish and nobody paid for a backup path, the site is a liability, not an asset.

I say no **before** deposit. Redirect honestly. Sometimes that means WordPress. Sometimes Shopify. Sometimes "keep your Facebook page and fix your Google Business listing first." All of those are better than a deposit on the wrong stack.

## The five questions I ask before I quote

Write these down if you're hiring anyone for a site - static or not:

1. **Who updates content, and how often?** Daily from five people is a different job than monthly from one owner.

2. **Anything private behind a login?** Members, clients, students, donors with accounts - name it early.

3. **Anything that changes in a live database?** Inventory, bookings, user-generated content, real-time feeds.

4. **Cart or checkout you expect me to host?** Not "we might sell stickers someday." Actual money moving on your domain.

5. **What happens if I'm unavailable for a month?** If the answer is "everything stops," we need a publish path that doesn't run through my inbox only.

**Two or more dynamic yes answers** → not this stack for me. One yes might be fine with an embed or an external tool carrying the dynamic part. Zero yes answers → we're probably talking.

## Updates that still fit static (when you're honest upfront)

Static doesn't mean "frozen forever." It means the publish path is deliberate.

**Owner learns Git.** Twenty-minute lesson: edit markdown, commit, push. A neighbor chose this, broke the build once on a typo, fixed it the same night. That's success.

**Retainer batches.** They email copy or photos; I push on a schedule. Priced per batch. Pastor sends a bulletin PDF every Friday - one push, no drama.

**Git-based CMS.** When Git won't land but scope still fits static. Buttons instead of terminal. Quoted separately from the build.

**Seasonal pages and light edits.** Annual camp schedule, holiday hours, a new staff photo. Fine on retainer or a short Git lesson.

**External embeds.** Donate links, Formspree forms, Calendly, booking widgets, delivery apps. The site stays static; the tool carries the moving part.

The pattern: **smallest site that solves the job**. Don't force Hugo where WordPress was the honest answer. Don't force WordPress where a single landing page and a Calendly link would do.

---

## Why saying no is part of the pitch

Clients hire judgment, not enthusiasm. Anyone can say yes. The person worth hiring tells you when the tool is wrong **before** money changes hands.

I've turned down work that would have paid. I've also kept relationships because the "no" came with a clear reason and a pointer to someone who fits better. That lands better than a half-built store and three months of patch-up invoices.

If you're reading this because you're evaluating me for a site: bring the hard questions. I prefer them in the first call, not in a panic email after launch.

If you're evaluating **yourself**: be honest about who publishes Tuesday afternoon. That answer picks the stack more reliably than any framework debate.
